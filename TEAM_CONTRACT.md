3. Conflict Plan

3.1 Process for resolving disagreement

When a disagreement arises (technical or interpersonal), the team will: (a) name the specific point of disagreement explicitly rather than letting it stay vague; (b) each teammate states their view in turn without interruption; (c) the team looks for a consensus decision within the same conversation; (d) if consensus isn't reachable, the team accepts whichever compromise every teammate can live with, and moves forward rather than staying blocked.

3.2 If one teammate is dominating or steamrolling

- First step: a direct, private, respectful conversation from any teammate who notices it, naming the specific behavior and its impact.
- Escalation: if it continues, the team raises it together at the next sync as a group process issue; if still unresolved, escalate per §3.5.

3.3 If one teammate is under-contributing or missing check-ins

- First step: a private, respectful, direct message checking in — assuming good intent and asking what's blocking them, not accusing.
- Escalation: if it continues after that conversation, the team raises it collectively at the next sync, and if still unresolved, escalates per §3.5.

3.4 Handling mismatched skill levels

More-experienced teammates coach by explaining reasoning and pointing to resources rather than taking over the keyboard/PR — pairing sessions favor "drive, don't take over." Less-experienced teammates are expected to ask for help after a fixed time-box (e.g., 30–45 minutes stuck) rather than disappearing into a rabbit hole alone.

3.5 Escalation path

If the team cannot resolve a conflict internally, a message is sent to the Leadership Team via Slack.

6.2 Branch strategy

- main: production-ready; all PRs land here after review.
- dev: feature-branch.
- Feature branches: feature/<slug> or <teammate-github>/<slug>. One task per branch. Delete branches after merge.

6.3 PR review workflow

Every PR to main requires 2 teammate approvals before merging (branch protection, no bypass). PR author does not merge their own PR. Stale-approval dismissal is on — new commits reset approvals. PR title format: <slug>: <short imperative summary>. PR body includes what changed, any teammate action needed, and test evidence (e.g., pytest -v output/screenshot).

6.4 Merge cadence

The team merges to main daily during sync plus on-demand for blocker unblocks. A teammate marks their PR ready in the Slack channel and tags a reviewer when it's ready to merge.

6.5 GitHub Project board — how your team runs it

- Columns: Backlog / In progress / In review / Done.
- Every task is a card with an owner; no unassigned cards in In progress.
- Automations: PR opened → In review; PR merged → Done.
- If a card sits in In progress for more than 24 hours without a note, it's raised at standup.
